Overview

Why Bitcoin for AI Agents?

Bitcoin was the first cryptocurrency and remains the largest by market capitalization. While newer chains like Ethereum and Solana offer programmability and DeFi, Bitcoin holds a unique position as digital gold â€” a store of value that has survived 15 years of market cycles, regulatory attacks, and technical challenges. For AI agents managing real-world value, the ability to hold, send, and receive BTC is foundational.

An AI agent managing a long-term treasury might hold a portion of assets in BTC as a non-correlated store of value. An agent acting as a service provider might accept BTC payments from users or other agents who prefer Bitcoin's security and broad acceptance. A cross-chain arbitrage agent might move value between the Bitcoin network and other chains when price differentials justify it.

Purple Flea's Bitcoin API provides a fully non-custodial, BIP-84-derived native SegWit wallet. Your agent gets a bc1... address that works with every major Bitcoin wallet and exchange. Transaction signing happens server-side with your encrypted private key; you always retain control of your mnemonic and can recover funds independently of Purple Flea at any time.

Address Types

Native SegWit by Default

Purple Flea generates native SegWit (bech32) addresses for all Bitcoin agent wallets by default. These addresses start with bc1q... and use the BIP-84 derivation path m/84'/0'/0'/0/0.

Native SegWit provides the lowest possible transaction fees on the Bitcoin network â€” roughly 40% cheaper than legacy P2PKH addresses and 25% cheaper than P2SH-SegWit addresses. For agents that transact frequently on Bitcoin, the fee savings are material.

P2SH-SegWit addresses (starting with 3...) are also supported for compatibility with older exchanges or wallets that do not yet accept bech32 addresses. Specify address_type: "p2sh-segwit" in your wallet configuration to use this format instead.

Bitcoin Fundamentals

The UTXO Model Explained

Bitcoin uses an Unspent Transaction Output (UTXO) model, which is fundamentally different from the account-based model used by Ethereum and Solana. Understanding this distinction helps agents reason about their Bitcoin holdings correctly.

In the UTXO model, your wallet does not have a single "balance" â€” it has a collection of discrete coins (UTXOs), each with a specific value and a reference to the transaction that created it. When you send Bitcoin, you select one or more UTXOs as inputs, pay the recipient and yourself (as change) as outputs, and the fee is the difference between inputs and outputs.

For agents, this means: transaction fees scale with UTXO count, not just transfer amount. An agent that receives many small BTC payments may accumulate many small UTXOs, making future transactions more expensive. Purple Flea automatically performs UTXO consolidation when your fee savings exceed the consolidation cost.

Fee Management

Automatic Fee Estimation

Bitcoin transaction fees are measured in satoshis per virtual byte (sat/vB). The required fee rate fluctuates with mempool congestion â€” during periods of high activity it can spike from 5 sat/vB to 200+ sat/vB. Purple Flea queries the current mempool fee rates in real time and selects the appropriate rate based on your fee_priority setting.

slow (next 6 blocks, ~60 min) â€” lowest fee, suitable for non-urgent transfers. standard (next 3 blocks, ~30 min) â€” default, good balance of cost and speed. fast (next block, ~10 min) â€” highest fee, use for time-sensitive transactions.

Replace-by-Fee (RBF) is enabled by default on all transactions. If a transaction is stuck in the mempool because the fee rate you paid is now insufficient, your agent can call POST /wallet/rbf with a higher fee rate to broadcast a replacement transaction and get faster confirmation.

Lightning Network

The Lightning Network is a layer-2 payment protocol built on top of Bitcoin that enables near-instant, near-free micropayments. While on-chain Bitcoin transactions settle in 10â€“60 minutes and cost $0.50â€“$20 in fees, Lightning payments settle in milliseconds and cost fractions of a cent.

Lightning is particularly exciting for AI agents because it enables the kind of high-frequency, low-value payment patterns that are impractical on-chain. An agent could stream micro-payments per API call, per token generated, or per compute unit consumed â€” enabling true pay-as-you-go micropayment economies between agents.

Code Example

Python Agent Example

An agent that monitors for incoming BTC payments and automatically converts to USDC on Ethereum.

btc_convert_agent.py
import requests import time import os API_KEY = os.environ["PURPLEFLEA_API_KEY"] BASE_URL = "https://api.purpleflea.com" HEADERS = {"X-API-Key": API_KEY, "Content-Type": "application/json"} def get_btc_balance() -> dict: r = requests.get( f"{BASE_URL}/wallet/balance", params={"chain": "btc"}, headers=HEADERS ) return r.json() def swap_btc_to_usdc_eth(btc_amount: float): """ Cross-chain: first send BTC, then swap WBTC to USDC on ETH. Here we use the built-in cross-chain bridge endpoint. """ r = requests.post( f"{BASE_URL}/wallet/bridge", json={ "from_chain": "btc", "to_chain": "eth", "from_asset": "BTC", "to_asset": "USDC", "amount": str(btc_amount), }, headers=HEADERS ) return r.json() def monitor_and_convert(): """ Watch for incoming BTC. When confirmed balance increases, convert the new BTC to USDC on Ethereum. """ initial = get_btc_balance() last_balance_sats = initial["balance_sats"] address = initial["address"] print(f"Monitoring BTC address: {address}") print(f"Starting balance: {last_balance_sats} sats") while True: time.sleep(60) data = get_btc_balance() current_sats = data["balance_sats"] if current_sats > last_balance_sats: new_sats = current_sats - last_balance_sats new_btc = new_sats / 100_000_000 print(f"Received {new_sats} sats ({new_btc:.8f} BTC)!") print(f"Converting to USDC on Ethereum...") result = swap_btc_to_usdc_eth(new_btc) print(f"Conversion result: {result}") last_balance_sats = current_sats if __name__ == "__main__": monitor_and_convert()
